Lab ISS | the project cautiousExplorer with Actors

Introduction

This case-study deals with the design and development of proactive/reactive software systems based on the concept of Actor, as introduced in LabIss2021 | wshttp support with ActorBasicJava observers.

Requirements

Design and build a software system that allow the robot described in VirtualRobot2021.html to exibit the following behaviour:
  • the robot lives in a rectangular room, delimited by walls that includes one or more devices (e.g. sonar) able to detect the presence of obstacles, including the robot itself;
  • the robot has a den for refuge, located in the position shown in the picture
  • the robot works as an explorer of the room. Starting from its den, the goal of the robot is to create a map of the room that records the position of the fixed obstacles. The presence of mobile obstacles in the room is (at the moment) excluded;
  • since the robot is 'cautious', it returns immediately to the den as soon as it finds an obstacle. It also stops for a while (e.g. 2 seconds), when it 'feels' that the sonar has detected its presence.

Requirement analysis

In addition to the definitions clarified in the lia_salvatore_ce.pdf document, another meeting with the client defines:
  • map: data structure that keeps track of the path taken by the robot and the obstacles found.
Regarding actions(verbs):
  • feels: the sonar emits a beam that when it detects the robot sends a signal.

Problem analysis

Relevant aspects

  1. Create a distributed system consisting of two macro-components:
    • the (virtual) robot supplied by the client
    • our application (cautiousExplorer) which sends commands to the robot in order to satisfy the requirements
  2. The robot can be controlled via the network in two different ways, as described in VirtualRobot2021.html: commands:
    • sending messages to port 8090 with HTTP POST protocol
    • sending messages to port 8091 with a websocket
  3. Since there are numerous libraries in many programming languages that allow the sending of these commands, no significant abstraction-gap is identified on an operational level.
  4. We estimate that a first prototype of the application should be able to be built in six working days (at most).

Logical architecture

 The exact nature of the cautiousExplorerActor component will be defined in the design phase.

Regarding the interaction we can say that:
  • the use of the HTTP protocol seems completely adequate, at least in a first phase;
  • the use of the websocket could prove to be more flexible (as it allows to receive information emitted by WEnv in a 'spontaneous' way) and more efficient (as it reduces the protocol hierarchy).

Another model (UML)

 This model 'abstracts' from the technological details of the interaction.

Problems identified

  1. Interaction abstraction

    The software system should be made as independent as possible from the communication protocol used for the interaction with WEnv.

  2. Testing

    In order to perform an automatable TestPlan, the application must be observable, that is, make its effects verifiable by a machine.
    In particular, it is necessary to explicitly represent the knowledge of the answers obtained by Wenv (sensors or cameras are not available)

    Within the application, a map capable of providing a model of the robot's actions must be created. In this way a test program can understand the map to perform the tests.

    We can prefigure that the solution of the problem consists of the following algorithm: 
    the robot starts from the den
until the robot hits an obstacle:
	send command to the robot
if the sonar detect the robot:
	the robot stop for 2 seconds
	resume robot
save the position of the obstacle
the robot returns to the den

Strategy chosen for the test plan

To make the discussion easier, we introduce the following command abbreviations cril: 
w : express the move {"robotmove":"moveForward", "time": 100}
s : express the move {"robotmove":"moveBackward", "time ":100}
l : express the move {"robotmove":"turnLeft ",   "time ":300}
r : express the move {"robotmove":"turnRight",   "time ":300}

Test plans

TESTPLAN 1:
for each successful move, we add the move identifier to a moves string (initially empty).
We set the time value for the w, s moves to move the robot in 'small steps' (robot-unit movements), so as to reach an obstacle with N> 1 moves.
When the robot hits an obstacle, the string must take the following form: 
moves: "[lr]*[ws]+[wsrl]*"     * : repeat 0 or more times
Then the robot returns to the den replacing the forward moves with the backward moves, the left moves with the right moves and vice versa. The return moves are saved in a second string. So the two strings must be the opposite of each other.
The robot's moves are organized so that you can rerun to facilitate resolution in case of problems.

Actor symbol:


  1. Now, in our analysis and design phase, we can talk of two main types of components: objects (POJO) and actors.

  2. An ActorBasicJava has the capability to interact with other actors (known by reference) by using the following local operations: 
    protected void forward(String msg, ActorBasicJava dest) { dest.send(msg); }
protected void request(String msg, ActorBasicJava dest) { dest.send(msg); }
protected void reply(String msg, ActorBasicJava dest) { dest.send(msg);   }


  3. An ActorBasicJava can be considered a sort of 'local-microservice'
